## Multi-User Isolation Hardening

- Implemented the first concrete multi-user isolation hardening pass after researching shared-schema SaaS patterns, PostgreSQL RLS, OpenFGA/ZITADEL authorization models, OWASP logging/key guidance, pgAudit, and Fastify rate-limit behavior.
- The app now has a deterministic personal tenant id derived from `User.id`, and `TenantContext` includes `tenantId` in addition to `userId/sourceId/sessionToken`.
- Authenticated rate limits no longer use raw session-token prefixes; they are tenant-scoped where possible and hashed-session-scoped as fallback.
- Redis session keys are now SHA-256 hashes of session tokens, with one-time compatibility for old raw-token keys.
- Added tenant audit logging under `apps/bff/data/audit-log/<hashed-tenant>/<month>.jsonl`, best-effort Prisma `AuditLog` mirroring, and `/api/security/audit-log` for the current user only.
- Added audit events for the main write surfaces: auth login/logout, mailbox source changes, Outlook connection, priority rules, notification preferences, KB exports/backfill/knowledge cards, and manual agent memory.
- KB artifact APIs/tools now return `mail-kb://documents/...` public paths instead of absolute filesystem paths, and export reports store relative paths.
- Disabled the legacy global `mail-kb-service.ts` singleton so flat shared KB storage cannot be re-enabled by env flag.
- Added a Prisma migration for `AuditLog` plus composite `MailSource(id,userId)` ownership guards on source-owned tables. RLS was intentionally left off because the runtime does not yet set `mery.user_id`/`mery.tenant_id` in every Prisma transaction.
- The bottom Dock was made smaller and glassier per browser diff feedback.
- Audit Round 1 found the premature RLS enablement High and the legacy KB escape-hatch Low; both were fixed. Round 2 returned clean. Evidence: `.harness/audit/2026-04-19-active-multi-tenant-isolation-hardening-audit.md`.
